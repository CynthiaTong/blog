Posted on 2018-12-11

Reading notes of Section 5.1, Algorithms 4th Edition by Robert Sedgewick and Kevin Wayne.

Key-indexed counting

A simple and efficient sorting method for key-indexed strings (and other objects). With small radix R, it is a linear-time (O(N)) sort, where R = # distinct keys, N = # strings.

Idea: Pre-allocate spaces for each key precisely (by computing the frequencies), then fill in each slot sequentially with the input strings.

Property: key-indexed counting is stable.

Steps:

  1. frequency counts
  2. counts => indices
  3. distribute specific data
  4. copy back to original string array

Note: this sorting method does not comply with the theoretical lower bound O(nlogn) because it is not based on comparison.

LSD (least-significant-digit first) radix sorting

To sort an array a[] of strings that each have exactly W characters, we do W key-indexed counting sorts: one for each character position, proceeding from right to left.

Real-world example: card dealing - Deal into 13 piles, pick up the piles in order, then deal into four piles => sorted deck.

Application: suitable for fixed-width strings (IDs, license plates, etc).

Running time: O(WN), linear in the # characters of the input.

LSD

Adaptation to variable-length strings: keep track of the MSD of each shorter string, and use them in subsequent passes of Key-indexed counting.

MSD (most-significant-digit first) radix sorting

Can be used directly for variable-length strings.
Similar to quicksort, but partitions into one subarray for each possible value of the first (leftmost) character, then recursively sort the subarrrays.

To sort an array a[] of strings, we sort them on their first character using key-indexed counting, then (recursively) sort the subarrays corresponding to each first-character value.

Modifications: because the strings are of variable size, need to have one index in the count array store the short strings at each iteration - len(count) = R + 2.

MSD

important issues:

  • importance of cutting off small subarrays! Avoid sorting for numerous subarrays of small sizes by using a slower sort, e.g. insertion sort. Otherwise, dramatic explosion in running time & space for larger radix, e.g. R = 65536 for Unicode.
  • Slow in the case of equal keys - key-index counting is expensive for sorting equal keys or keys with long common prefixes.
  • Extra Space of the auxiliary count, aux arrays. aux can be moved out of the recursive function by sacrificing stability; but count cannot be moved out.
  • Worst case linear running time, in the case of all equal strings. Sub-linear in the random string model.

Three-way string quicksort

characteristics:

  • Appropriate when strings have long stretches of equal data - avoid the large number of empty subarrays in MSD.
  • Does not use extra space.

5. Replication: weak consistency model protocols

  • Strong Consistency is achieved by enforcing global total order - undesirable in case of unavailability due to partition, and expensive b/c the majority of nodes must be contacted (more than once, think 2PC).

  • Eventual Consistency with:

    • probabilistic guarantees, ex. Amazon Dynamo
    • strong guarantees, CRDT (convergent replicated data types), CALM (consistency with logical monotonicity) conjecture

  • Dynamo

    • weak consistency guarantee + high availability
    • k/v store with consistent hashing.
    • uses sloppy (partial) quorums (rather than strict, majority quorums)
    • user specify W / R: # nodes queried for write / read.
      Usually: W + R > N = 3 to have at least 1 overlapping node - less likely to return a stale value
    • probabilistic “strong consistency” guarantee. Upon node failure, Dynamo replaces failed nodes with unrelated but accessible nodes.
    • Conflict detection & read repair
      • metadata associated with each piece of data
      • metadata types: no metadata (slowest wins), timestamp, version number, vector clock

  • Replica Synchronization

    • Gossip: every timeframe t, nodes have prob p of attempting to synchronize - no guarantees.
    • Merkle Tree to make synchronization efficient

  • PBS (probabilistically bounded staleness)

    • estimates degree of inconsistency using simulation & real-world system data

  • CRDTs

    • certain set of operations are commutative - same result in any order
    • formally: the operations need be:
      • 1) Associative, so grouping doesn’t matter
      • 2) Commutative, so order doesn’t matter
      • 3) Idempotent (a+a=a), so duplication doesn’t matter
    • Mathematically called join semilattices and meet semilattices. Ex: max, union.

  • CALM theorem

    • programming models in which the order of statements doesn’t play a significant role
    • Order-independent Declarative languages, Ex. Map Reduce, SQL
    • CALM: logically monotonic programs are guaranteed to be eventually consistent & safe to execute without coordination
    • Monotony: if sentence φ is a consequence of a set of premises Γ, then it can also be inferred from any set Δ of premises extending Γ
    • In plain words, no conclusions can be invalidated by learning new knowledge
    • relational algebra (SQL), Datalog
    • open-world assumption (OWA): we can only safely assert something we can deduce from what is known. What is currently not known to be true is unknown, not true or false. OWA applies when a system has incomplete information.
    • closed-world assumption (CWA): what is not currently known to be true, is false. CWA applies when a system has complete information (such as databases).
    • logical circumscription: the known entities are all there are. No additional information to invalidate our assertion, in case of non-monotonic logic.

    • in order to handle nonmonotonicity one needs to either use distributed coordination to ensure that assertions are made only after all the information is known or make assertions with the caveat that the conclusion can be invalidated later on.

    • CALM application: the Bloom language.

Intuition

The naive approach matches B starting from each character in A. This simple approach works well when B is very short. The running time will be O(mn), approximately O(n) when m is small. However, this running time is too long if we have relatively long B, and/or the alphabet is small with lots of repeating patterns.

The intuition of KMP is: when a mismatch is found at A_i and B_j, we want to avoid going back to p (the last matching start position) + 1. If we do go back every time, we waste computation matching characters we have already tried to match before.

Bit Manipulation (1) Bit Operators & Hamming Distance

Posted on 2018-01-19

This post is the beginning of a series of articles on bit manipulation and the programming problems it solves.

Topics covered:

  • bitwise operators
  • hamming weight and hamming distance problems

Bit Operators

This is mostly recap from the Computer Systems Organization class I took last fall. The basic definition of bit operators can be found on Wikipedia. I divide the operations into two categories: operations on one single bit, and bit shifts. Things we can do with a single bit include: AND, OR, XOR, and NOT. There are only two types of bit shifts: left shift and right shift.

Here are some handy tricks and rules from CtCI:

XOR(^) AND(&) OR(|)
x ^ 1s = ~x x & 1s = x x | 1s = 1s
x ^ 0s = x x & 0s = 0 x | 0s = x
x ^ x = 0 x & x = x x | x = x

It is important to note the difference between the bit operators &, |, ~, and the logical operators &&, ||, !.
For instances, 2 & 1 = 10 & 01 = 0, but 2 && 1 = 1; ~4 = ~100 = 11 = 3, but !4 = 0.
Rule is: when && with 0, the result is always 0, when || with nonzero values, the result is always 1; when ! with nonzero values, the result is always 0. In cases of the bitwise operations, however, the result varies.

Bit Shifts

Relatively simple concept. The left shift, denoted by <<, shifts the value left by 1 bit, effectively multiplying that number by 2. The arithmatic right shift >> does the contrary - it divides the number by 2. The other type of right shift is called logical right shift (denoted as >>> in Java). The only difference between arithmatic and logical right shifts is, arithmatic <=> preserving the MSB (Most Significant Bit); logical <=> padding the MSB with 0s. It really makes a difference when dealing with negative integers, since their MSB is always 1.

Hamming Weight & Hamming Distance

Definitions from Wikipedia:

The Hamming weight of a string is the number of symbols that are different from the zero-symbol of the alphabet used. It is thus equivalent to the Hamming distance from the all-zero string of the same length. For the most typical case, a string of bits, this is the number of 1s in the string, or the digit sum of the binary representation of a given number and the ℓ₁ norm of a bit vector.

In short,
hamming weight = #1s in the binary representation of a number
hamming distance between A and B = # of different bits in the binary representations of A and B = hamming weight of (A^B)

Another interesting problem on LeetCode is calculating the “total hamming distance” of an array of integers. The brute force solution is to use the hammingDistance function we discussed on every unique pair of integers in the array, which takes O(n^3) time - a nightmare for large inputs. Luckily, we have a workaround. A much better solution involves considering the array of integers as a whole. For each bit in the 32-bit integers, calculate the number of values in the array that has a 1 on that bit. Suppose this count is k for a certain bit, then given a total of n integers, there are (n-k) of them that have a 0 instead. Therefore, this particular bit contributes k*(n-k) to the total hamming distance.
